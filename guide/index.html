Two types of work — kept separate

🐛 Bug / QA Feedback

Something broken in the existing app. Raise as a GitHub Issue using the Functional or UI template. Ticket goes into the QA team in Linear. Never inside a project.

🚀 New Features

Work that doesn't exist yet. Lives inside one of the active projects: Try Before You Buy, WhatsApp Integration, or Master Admin. Same GitHub branch + PR convention applies.

💡
The one rule: If it's broken → GitHub Issue → QA team. If it's being built → Project ticket. Never mix the two.
StackBack · For QA / PM / Design

How to raise
a ticket

You don't need a Linear account. You don't need to know the label names. Just go to GitHub, fill in the form, and submit. Everything else is automatic.

Step by step
01
QA / PM / Designer
Go to GitHub → New Issue
Open the StackBack repo and click New Issue
▼
1
Go to: github.com/Sachin-Ulabs-Git/stackback-qa-demo/issues/new/choose
2
You'll see two options — pick the right one:

🐛 Functional / Module Issue — something is broken or doing the wrong thing
🎨 UI / Design Issue — something looks wrong or is misaligned
💡
Blank issues are disabled. You must use a template — this ensures every ticket has the right information for the developer.
02
QA / PM / Designer
Fill in the form
Dropdowns handle the labels — you just describe what's wrong
▼
1
Update the title — replace the placeholder with the actual issue:

Functional: [Purchase Error] Pricing shows $12 not $12.49 — Storefront Surface — Everpure
UI: [UI Feedback] Widget selector text clipped on mobile — Storefront Surface — Mobile
2
Fill the dropdowns — Error Type, Surface, Severity, Store. These become Linear labels automatically. You don't need to know Linear label names.
3
Describe what went wrong — write it naturally. Include steps to reproduce and attach screenshots. The more detail, the faster the dev can fix it.
✅
Good description: "In Portal Surface > Edit Drawer, while adding a freebie and swapping a product at the same time, the checkout link fired before the freebie was saved. Expected: save completes first. Steps: 1. Open edit drawer 2. Swap product 3. Add freebie 4. Click save."
03
Automatic
Ticket appears in Linear ● Auto
Within 30 seconds — no action needed from you
▼
1
A GitHub Action runs automatically and creates the ticket in Linear.
2
The ticket lands in Triage with all 3 labels already applied: surface label + error type label + severity label.
3
You'll see a link to the Linear ticket in the GitHub issue. That's it — your job is done. The dev will pick it up from Triage.
Severity — how to pick the right one
SeverityWhen to useExamples
CriticalCheckout, payment, or core user action is completely broken. Fix immediately.Cart not loading, payment failing, subscription can't be created
HighSignificant feature broken or degraded. Workaround exists but it's painful. A client has called about it.Pricing shows wrong values, edit drawer not saving
MediumNoticeable issue, limited user impact. No client complaint. Workaround exists.Order summary stale, portal settings not updating live
LowCosmetic or minor UX gap. No functional impact. Spotted internally.Button padding off, text clipped on rare screen size
💡
Quick rule: Client called or messaged about it → at least High. Checkout or payments involved → Critical. Spotted it yourself and it looks minor → Medium or Low.
When to use Backlog instead of Triage
📥
QA owns the Backlog decision. Move a ticket to Backlog only if all three apply: (1) cosmetic or non-blocking — no user flow is broken; (2) no customer has reported it; (3) does not affect checkout, payments, or scheduling. If in doubt, leave it in Triage for the PM to decide.
StackBack · For Developers

Pick up a ticket
and close it

From seeing a ticket in Triage to watching it close itself on merge. Two clicks in Linear, the rest is git and GitHub.

The full cycle
01
Dev · Linear · 30 seconds
Claim the ticket ● Manual
The only thing you do in Linear
▼
1
Open Linear → StackBack QA → Triage. Find your ticket. Read the full description — surface, error type, steps to reproduce, screenshots.
2
Move the ticket to Confirmed. This is the only manual state change in the entire flow.
3
Set yourself as Assignee. This is how the PM tracks who is working on what.
4
In the ticket sidebar → Development → click Copy git branch name. You'll get something like: tes-8-checkout-link-fires-before-freebie-saves
02
Dev · Terminal
Create the branch and fix the code
Always branch off main — never work on main directly
▼
1
Pull latest main and create your branch:
# Get the latest code git checkout main git pull origin main # Create branch using the name from Linear git checkout -b tes-8-checkout-link-fires-before-freebie-saves
2
Make your fix. Test locally. Then commit and push:
git add . git commit -m "fix: save freebie before generating checkout link — Closes TES-8" git push -u origin tes-8-checkout-link-fires-before-freebie-saves
03
Dev · GitHub
Open the Pull Request
One rule: include Closes TES-XX in the description
▼
1
Go to the GitHub repo. Click Compare & pull request on the yellow banner (or Pull Requests → New).
2
Set base: main, compare: your branch. Fill the title and description.
3
In the description, include the closing keyword — this is what triggers the auto-close:

Closes TES-8  or  Fixes TES-8  or  Resolves TES-8
4
Click Create pull request. Two things happen automatically:
• PR Guard checks for the Linear reference and posts a confirmation comment
• Linear auto-moves the ticket from Confirmed → In Review
⚠️
Forgot Closes TES-XX? PR Guard will post a warning comment. Edit the PR description to add it — the check updates automatically. Without it, the ticket won't auto-close on merge.
04
PM / Dev · Review
PR reviewed and approved
Someone reviews the diff before it goes live
▼
1
Reviewer looks at the diff — what changed, what was added or removed. Can comment on specific lines.
2
If changes needed, push more commits to the same branch. PR updates automatically — don't close and reopen.
3
Reviewer approves. Dev or reviewer clicks Merge.
✓
Automatic
Merge → ticket closes itself ● Auto
You don't touch Linear — the merge does it
▼
1
Click Merge pull request → Confirm merge on GitHub.
2
Within 5–10 seconds, Linear automatically moves the ticket to Done.
3
Delete the branch — it's done its job:
git push origin --delete tes-8-checkout-link-fires-before-freebie-saves
✅
Tested and confirmed: The PR merge → Linear auto-close cycle works. TES-5 moved to Done within 10 seconds of merging.

StackBack · Rules & Safeguards

The rules that
keep it working

The automation is only as good as the habits around it. These are the things that break the flow when they go wrong.

5 rules to never break
  • ❌
    Never manually close a ticket — the PR merge does it. Manual closures break code traceability and audit history.
  • ❌
    Never raise a ticket without using a GitHub template — blank issues skip the label automation. The ticket lands in Linear with no labels and can't be tracked or filtered.
  • ❌
    Never open a PR without Closes TES-XX — the auto-close won't fire and the ticket stays open forever. PR Guard will warn you if you forget.
  • ❌
    Never put feature work in the QA team — new features belong in a project (Try Before You Buy, WhatsApp Integration, or Master Admin). The QA team is bugs only.
  • ❌
    Never leave Triage tickets unactioned for more than 7 days — the Triage Hygiene Bot runs daily and will post a comment on stale tickets. Act on them before it fires.
  • ✅
    Always assign the ticket to the dev working on it — not the person who filed it. Set Assignee when you move to Confirmed. This keeps accountability clear.

For QA / Reporter / Designer

You found a bug.
Here's exactly what to do.

You don't need a Linear account. You don't need to know label names. Just open GitHub, fill in a form, and submit. Everything else is automatic.

Your job in 3 steps
1
You → GitHub
Open a new issue using the right template
Go to the Issues tab on GitHub. Choose Functional Issue or UI Issue. Never click "Open a blank issue."
2
You → GitHub form
Fill in all the dropdowns and describe what's broken
Update the title. Select Error Type, Surface, Severity, Store from the dropdowns. Write what happened and how to reproduce it. Add screenshots.
⚡
Automation — nothing you do
Ticket appears in Linear Triage within 30 seconds
Labels are applied automatically. The PM sees it in Triage. Your job is done. You'll see a link to the Linear ticket in the GitHub issue.
Step-by-step guide
01
Go to GitHub Issues
Open the StackBack repo and click New Issue
You'll see a template chooser — two options will appear
▾
1
Go to github.com/Sachin-Ulabs-Git/stackback-qa-demo/issues/new/choose
or open the repo → click Issues tab → click New issue
2
Choose the right template:

🐛 Functional / Module Issue — something is broken or doing the wrong thing (calculation wrong, action not firing, data not saving)

🎨 UI / Design Issue — something looks wrong (misaligned, wrong colour, clipped text, wrong sizing)
💡
"Open a blank issue" is disabled. You must use a template. This is enforced — it protects developers from receiving tickets without enough information to act on.
02
Update the title
Replace the placeholder with the real issue title
Follow the format — it becomes the Linear ticket title
▾
1
The default title is a placeholder. Replace it with the actual issue.

Functional format:
[Error Type] What broke — Surface — Store Name

UI format:
[UI Feedback] What looks wrong — Surface — Device/Store
2
Good examples:
[Purchase Error] Checkout link fires before freebie saves — Portal Surface — Zama
[Timeline Error] Order summary shows stale values after edit — Portal Surface — Everpure
[UI Feedback] Widget selector text clipped on iPhone 13 — Storefront Surface — Mobile
⚠️
Don't leave the placeholder title. "Error Type] What broke — Surface — Store" is not a valid ticket title. The PM and dev need to understand the issue from the title alone.
03
Fill the dropdowns
Select Error Type, Surface, Severity, and Store
These become Linear labels automatically — you don't need to know Linear
▾
1
Surface / Module
Storefront Surface
Portal Surface
Landing Surface
Order Module
CX Module
Selling Formats / Purchase Options
Customer Notification Module
Admin Notification Module
Settings & Config
Onboarding
Customer Module
Integrations Module
2
Error Type
Purchase Error — checkout, cart, pricing

Timeline Error — order summary, edit state

Scheduler Error — delivery job timing

CX Module Error — portal / widget settings

UI Feedback — design / visual issue
3
Severity
Critical — checkout fully broken

High — significant feature broken

Medium — noticeable, workaround exists

Low — cosmetic, no impact
ℹ️
All 3 labels are required. A ticket missing any of them can't be properly triaged or filtered by the PM. The hygiene bot will flag it if they're missing.
04
Write the description
Describe what went wrong and how to reproduce it
The more detail here, the faster it gets fixed
▾
1
What went wrong — use this pattern:

"In [Surface > Sub-area], while doing [action], [what happened]. Expected: [what should happen]."

Example: "In Portal Surface > Edit Drawer, while adding a freebie and clicking Save, the checkout link fired instead of saving the freebie. Expected: the freebie to be saved and the drawer to close."
2
Steps to reproduce — numbered list, specific enough that someone who wasn't there can reproduce it:

1. Go to Portal > [store name] subscription
2. Open the edit drawer
3. Click Add Freebie → select an item
4. Click Save
3
Screenshots — always attach if possible. Drag and drop directly into the text area. If there's an error message or console log, paste it too.
✅
Good description example: "In Storefront Surface > Pricing, the item total shows $12 instead of $12.49. Steps: 1. Open store, 2. Add item to cart, 3. View cart total. The decimal is being rounded. Screenshot attached."

▾

How to pick the right severity
Critical
A user-facing flow is completely broken. Fix needed now.
Checkout not loading
Payment not processing
Subscription can't be created
Cart keeps emptying
High
Significant feature broken. Workaround exists but it's painful. Client noticed.
Edit drawer not saving
Pricing shows wrong totals
Freebie not applying
Client messaged about it
Medium
Noticeable issue but limited impact. No client complaint. Workaround available.
Order summary stale
Portal settings lag on live
Notification timing off
Spotted internally
Low
Cosmetic or minor UX gap. No functional impact whatsoever.
Button padding off
Text clipped on rare size
Icon colour mismatch
Label slightly wrong
💡
Quick rule: Client called or messaged about it → at least High. Checkout or payment involved → Critical. You spotted it yourself and no user is blocked → Medium or Low. When in doubt, go one level higher — the PM can downgrade.
Do this / not this
✓ Do this
  • ✅ Use the GitHub Issue template every time
  • ✅ Replace the placeholder title with a real description
  • ✅ Fill all dropdowns — all 3 are required
  • ✅ Describe what happened AND what you expected
  • ✅ Add numbered steps to reproduce
  • ✅ Attach screenshots or error logs
  • ✅ When in doubt on severity, pick one level higher
✗ Not this
  • ❌ Don't DM the dev directly about a bug — it won't be tracked
  • ❌ Don't raise issues in Slack or Notion — they won't sync to Linear
  • ❌ Don't click "Open a blank issue" — it bypasses all automation
  • ❌ Don't leave the placeholder title unchanged
  • ❌ Don't skip the "Steps to reproduce" section
  • ❌ Don't raise new features here — only bugs and UI issues
⚙️ Developer
For Developers

Ticket confirmed.
Here's your full playbook.

From picking up a ticket in Linear to watching it close itself on merge. One manual click in Linear — the rest is git and GitHub.

Your full cycle
01
Linear · 1 minute
Claim the ticket ● Your one manual step
Open Linear. Move to Confirmed. Assign yourself. Copy branch name.
▾
1
Open Linear → StackBack QA team → Triage. Pick a ticket to work on. Read the full description — surface, error type, steps to reproduce, screenshots.
2
Move the ticket state to Confirmed. This is the only manual state change in the entire flow — everything else is automated.
3
Set Assignee to yourself. This is how the PM tracks who is working on what. An unassigned Confirmed ticket looks like nobody owns it.
4
In the ticket sidebar, look for the Development section → click Copy git branch name. You'll get something like:
tes-8-checkout-link-fires-before-freebie-saves

This is the branch name you'll use in the next step.
💡
Why this matters: The branch name from Linear links the branch to the ticket. It's how Linear knows which PR to move when you open it. Always use the branch name from Linear — don't make up your own.
02
Terminal · Git
Create the branch and write the fix
Always branch off main — never work on main directly
▾
1
Pull the latest main and create your branch with the name you copied from Linear:
# Always start from a fresh main git checkout main git pull origin main # Create and switch to your branch (paste the name from Linear) git checkout -b tes-8-checkout-link-fires-before-freebie-saves
2
Make your fix. Test it locally. When you're done, commit your changes:
# Stage your changes git add . # Write a clear commit message git commit -m "fix: save freebie before generating checkout link" # Push to GitHub (first time on this branch) git push -u origin tes-8-checkout-link-fires-before-freebie-saves
3
For subsequent commits on the same branch, just use git push (no need for -u origin branch-name again).
03
GitHub · Open PR
Open the Pull Request with Closes TES-XX in the description
This one line is what triggers the entire auto-close chain
▾
1
After pushing, GitHub shows a yellow banner: "Compare & pull request". Click it. Or go to Pull Requests → New pull request.
2
Set base: main ← compare: your branch. Write a clear title describing what was fixed.
3
In the PR description, include the closing keyword on its own line:

Closes TES-8   or   Fixes TES-8   or   Resolves TES-8

This is what triggers Linear to auto-close the ticket when the PR is merged.
github.com · stackback-qa-demo · pull requests · new
PR Title
fix: save freebie before generating checkout link (TES-8)
Description
The checkout link was firing immediately when the save button was clicked, before the freebie save promise resolved. Fixed by awaiting the save before generating the link.

Closes TES-8
GA
github-actions[bot] · just now
✅ PR Guard — Linear ticket linked

Great — this PR is linked to TES-8.
When this PR is merged, Linear will automatically move TES-8 to Done.
⚠️
If you forget Closes TES-XX: PR Guard posts a warning. Edit the PR description to add it — the check re-runs automatically and updates the comment. Don't close and reopen the PR.
04
Automatic · Linear
Ticket moves to In Review ● Auto
You don't need to touch Linear — PR opening does it
▾
1
As soon as the PR is opened, Linear detects it via the GitHub integration and automatically moves the ticket from Confirmed → In Review.
2
The ticket now shows a link to the GitHub PR in the Linear sidebar. The PM can see the PR status directly in Linear.
05
GitHub · Code Review
PR reviewed — address feedback if any
Don't close and reopen — just push more commits to the same branch
▾
1
Reviewer looks at the diff and leaves comments. Address any feedback by making changes on your branch and pushing again — the PR updates automatically.
2
Don't create a new PR for review feedback — push to the same branch. GitHub shows a timeline of all commits added after the review.
3
Once approved, merge the PR.
✓
Done · Automatic
Merge → ticket closes itself in Linear ● Auto
You don't need to update Linear — the merge does everything
▾
1
Click Merge pull request → Confirm merge on GitHub.
2
Within 5–15 seconds, Linear automatically moves TES-8 → Done. No manual update needed. The PM sees it immediately.
3
Clean up the branch:
# Switch back to main and pull the merged code git checkout main git pull origin main # Delete the local branch — it's done its job git branch -d tes-8-checkout-link-fires-before-freebie-saves # Delete the remote branch on GitHub git push origin --delete tes-8-checkout-link-fires-before-freebie-saves
✅
Confirmed working. TES-8 moved to Done within 10 seconds of PR merge. The full cycle from Triage → Done takes zero manual Linear updates after the initial Confirmed click.

📋 Lead / PM
For Lead / PM

New tickets are in Triage.
Here's how to work them.

The automation handles labelling and state transitions. Your job is to review what's in Triage each day, make the call on priority, assign to the right dev, and keep the board clean.

Your daily workflow
9:00 AM · Weekdays
Triage Hygiene Bot runs automatically
Checks for stale tickets (>7 days) and missing labels. Posts a comment directly on any offending ticket in Linear. GitHub Actions also shows a summary report.
Morning · Daily
Review Triage — make a call on every new ticket
Open Linear → StackBack QA → Triage. Every ticket here needs one of three actions: Confirm it (dev picks it up), park it to Backlog, or close it if it's not valid.
Ongoing
Monitor In Review → Done pipeline
Tickets in In Review have a PR open on GitHub. On merge they auto-move to Done — you don't need to do anything unless a PR has been sitting open for too long without being merged.
Weekly
Board hygiene check
Review the Backlog — close tickets that are no longer relevant. Check Confirmed — if something has been Confirmed for more than 3 days with no PR, follow up with the dev.
The triage decision — what to do with each ticket
Decision tree — for every ticket in Triage
Is this ticket valid and actionable?
✓ Yes
Now: is a dev available to pick it up?
✗ No — not valid
→ Close it
Add a comment explaining why before closing (e.g. "Expected behaviour — not a bug")
Is a dev available and should this be fixed now?
✓ Yes — fix this sprint
→ Move to Confirmed
Assign to the dev. Add a priority label if not already set. Dev will branch and start work.
~ Not now — low priority
→ Move to Backlog
Only if cosmetic / non-blocking / no client impact. Leave a comment with the reason.
📌
The Backlog rule: A ticket can only go to Backlog if ALL three apply: (1) cosmetic or non-blocking — no user flow is broken; (2) no customer has reported it; (3) does not affect checkout, payments, or scheduling. If any of these don't apply, keep it in Triage or Confirm it.

Stale ticket alert — when the bot fires
🤖
Bot · 9am daily
Hygiene check runs automatically
Checks every ticket currently in Triage. Flags any that have been there more than 7 days, or are missing required labels.
💬
Bot → Linear
Comment posted on the offending ticket
The bot posts: "⏰ This ticket has been in Triage for X days. Action needed: move to Confirmed, Backlog, or close." You'll see it in the ticket's activity feed.
📋
You → Action required
Make a decision on the stale ticket
Move to Confirmed (assign to a dev), move to Backlog (if low priority), or close it (if no longer valid). Don't leave it in Triage once flagged.
🚨
The GitHub Action also fails when there are stale or unlabelled tickets — this makes it visible in the Actions tab. It's not a deployment failure — it's an intentional alert. Resolve the tickets to clear it.
QA team vs projects — what goes where

🐛 StackBack QA team

Bugs and UI issues only. Something that was working and now isn't, or something that's wrong in the existing product. Raised via GitHub Issues. Triaged here. Never put feature work in this team.

🚀 Projects (feature work)

New things being built. Try Before You Buy, WhatsApp Integration, Master Admin. These live inside the relevant project in Linear, not in the QA team. Same GitHub branch + PR convention applies.

💡
Quick test: "Is something broken that used to work?" → QA team. "Is this something new being built?" → a project. Never both.
